## 2. Phase 2B Pre-Implementation Risks

Phase 2B should be treated as higher risk than Phase 2A because it will begin touching stock-changing behavior.

Key risks:

- Existing legacy QT data is not sufficient to reconstruct full historical inventory lifecycle.
- The 60 paid-but-unfulfilled legacy candidates are not confirmed operational reservations.
- Legacy import/sync scripts can still overwrite QT order tables if run without cutover guards.
- Any stock-changing API must use transaction boundaries and row locks to prevent concurrent over-reservation or over-deduction.
- `qt_inventory_monthly` currently exists, but `qt_inventory_reservations` and `qt_inventory_transfers` are not implemented.
- Existing `qt_inventory_movements` is partially extended for Phase 2A, but it is not yet a complete Phase 2B reservation/fulfillment ledger.
- Existing order status values are still legacy-oriented and must not be renamed or remapped casually.
- LINE User identity must not be treated as a formal member identity; QT order ownership must resolve through Pastoral Member mapping.

## 3. Current Invariant Enforcement

Current invariant:

```text
physical_quantity = reserved_quantity + retail_quantity
```

Current enforcement layers:

1. Database constraint on `qt_inventory_monthly`:
   - `physical_quantity >= 0`
   - `reserved_quantity >= 0`
   - `retail_quantity >= 0`
   - `physical_quantity = reserved_quantity + retail_quantity`
   - `qt_month >= '202609'`
   - `qt_type IN ('ADULT', 'CHILD')`

2. API validation in `api/src/modules/qt/routes.js`:
   - `qt_month` is normalized and validated as `YYYYMM`.
   - `qt_type` is normalized to `ADULT` / `CHILD`.
   - non-negative integer quantities are required.
   - `reservedQuantity + retailQuantity` must equal `physicalQuantity`.

3. UI behavior:
   - users select year and month instead of typing a free-form QT month.
   - retail quantity is derived from physical minus reserved in the monthly inventory modal.

Current limitation:

- Phase 2A creates the stock master and guards manual monthly inventory creation.
- It does not yet update reserved or physical stock from payment approval, cancellation, refund, fulfillment, Line Bot ordering, or cross-church pickup.

## 4. Recommended Phase 2B Scope

Recommended Phase 2B should focus on reservation foundation only, before fulfillment and cross-church transfer.

Suggested in-scope items:

1. Create DBA-approved reservation schema if still accepted:
   - `qt_inventory_reservations`
   - additive references from `qt_inventory_movements` to reservation/order/order item if needed.

2. Add reservation service logic:
   - create reservation only when a QT order becomes confirmed paid.
   - release reservation for cancellation/refund paths only after the payment state design is confirmed.
   - keep all writes in a database transaction.
   - lock `qt_inventory_monthly` with `SELECT ... FOR UPDATE`.

3. Add audit and inventory log writes:
   - every reservation and release must append inventory/audit evidence.

4. Add read-only reservation visibility:
   - show reserved quantity and related reservation rows for admin reconciliation.

5. Add manual reconciliation tooling only if DBA and operations approve:
   - export paid-unfulfilled candidates.
   - do not auto-import unconfirmed legacy candidates.

## 5. Phase 2B Out Of Scope

Phase 2B should not touch these areas unless a separate approved task explicitly expands scope:

- QT pickup fulfillment.
- Cross-church pickup.
- `qt_inventory_transfers` implementation.
- Line Bot / LIFF QT ordering.
- forecast engine.
- broad order state-machine rewrite.
- automatic backfill of the 60 legacy paid-unfulfilled candidates.
- mutation of 2026-08 or earlier legacy months into the new invariant model.
- administrative supply inventory logic.
- DNS, NAS firewall, reverse proxy, LINE Developer Console, or Apps Script deployment settings.
